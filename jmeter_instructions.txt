Instrucciones JMeter y variantes cache frio/caliente:

1) Obtener Load-tests.jmx
   - En el repo: /Jmeter-test/Load-tests.jmx
   - Copia ese archivo al directorio donde ejecutarás run_jmeter_docker.sh

2) Actualizar el host:
   - Abre el .jmx en JMeter GUI (local) y en HTTP Request Defaults o Request setea:
       - Server Name or IP -> REPLACE_WITH_ALB_DNS (o el DNS real)
       - Port -> 80
   - O usa sed: sed -i "s/REPLACE_WITH_ALB_DNS/alb-dns-here/g" Load-tests.jmx

3) Plan de carga (sugerencia para tu ASR original: ramp-up 50->100 usuarios)
   - El .jmx del repo tiene varios Thread Groups (GET y POST). Ajusta Number of Threads y Ramp-up para:
       - Fase de test: número threads = 100, ramp-up = 60-120 s (según tu plan)
       - Loop Count: loops suficientes para generar tráfico (o duración con Scheduler)

4) Ejecución no-GUI (CloudShell con Docker):
   - export ALB_DNS="tu-alb-dns"
   - ./run_jmeter_docker.sh

5) Cold cache (cache frío):
   - Reinicia las instancias (systemctl restart monitoringapp.service o reinicia EC2) antes de la prueba.
   - Ejecuta JMeter inmediatamente.

6) Hot cache (cache caliente / pre-warm):
   - Ejecuta un pre-warm: curl -s "http://ALB_DNS/measurements" en bucle o ejecuta JMeter con 5-10 threads durante 1-2 minutos.
   - Ejecuta la prueba principal después.

7) Métricas y criterios:
   - Exporta results.jtl y abre HTML report (report folder). Saca screenshots del Summary Report.
   - Verifica:
       - %Error <= (según tu ASR: 1% o 10% del ejercicio)
       - Avg Time <= 1000 ms
       - 95th percentile <= 1000 ms
       - Throughput (req/s)
       - Latency (server processing time) — JMeter report y CloudWatch TargetResponseTime

8) Recomendación final:
   - Repite pruebas varias veces.
   - Captura resultados en la tabla del laboratorio y saca capturas.
